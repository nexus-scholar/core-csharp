Module documentation

Protocol

Scientific protocol lifecycle records, approvals, waivers, amendments, and immutable protocol content for review and workflow execution.

Scope and ownership

Protocol owns in-kernel scientific records for study plan intent, required decision declarations, draft edits, and immutable approved versions. It depends only on NexusScholar.Kernel and does not define storage, API schema, CLI behavior, or UI concerns.

  • Draft lifecycle: draft → ready_for_review → approved and superseded/withdrawn.
  • Decision integrity: no decision overwrite once recorded; duplicate keys are rejected by explicit error category.
  • Approval and approval policy enforcement for human-only authority and digest binding.
  • Canonical digest contract for protocol content and approval records using Kernel scopes.

Gate and ADR status

GATE-03 is accepted for local protocol lifecycle behavior. This is bounded by ADR 0003 and ADR 0004.

  • Protocol content digest uses DigestScope.ProtocolContent with schema nexus.protocol-content.
  • Approval record digest uses DigestScope.ApprovalRecord with schema nexus.protocol-approval-record.
  • Approval is restricted to human actors; stale content binds, duplicate approvals, blocking unresolved decisions, and dual-policy failures are rejected with stable categories.
  • Evidence bundle: GATE-03-EVIDENCE.

Related modules (consumers and boundaries)

  • Workflow: compiles approved protocol versions and validates waivers/invalidation plan bindings.
  • Bundles: binds protocol versions into review manifests.
  • Provenance: records protocol binding metadata from protocol versions.
  • CLI: consumes protocol records in sample entry paths.
  • AI: no direct dependency from Protocol; approvals remain local scientific authority rules.

Current issues to track

  • Draft overload convenience surface includes hard-coded legacy defaults. ProtocolDraft.Create(ids, title, requiredDecisionKeys) still emits fixed placeholders (legacy template id and project id), so it should remain sample/utility only.
  • Draft identity semantics are confusing. ProtocolDraft.Id mirrors ProtocolId rather than exposing a single stable draft identifier; callers should use DraftId explicitly for draft-level identity.
  • Withdrawal workflow remains scoped. Current behavior validates human-only authority and staleness paths but does not expose a dedicated high-level withdrawal action surface in this contract.
