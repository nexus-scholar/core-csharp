# Golden Fixture Plan

Status: gate planning fixture catalog. No fixtures are generated by this document.

## Rules

- Generate fixtures only from the PHP commit pinned in `specs/SOURCE.lock.json` when the source is PHP behavior.
- Never hand-edit generated outputs.
- Record source commit, generator command, generator version, input digest, and output digest for every generated fixture set.
- Treat a C# test failure as a defect or a conflict signal, not permission to replace the fixture.
- Classify every difference as one of:
  - equivalent serialization,
  - intentional change,
  - PHP defect,
  - C# defect,
  - unresolved specification conflict.

## Storage Plan

- Blueprint and contract fixtures: `fixtures/conformance/`
- PHP-generated differential fixtures: `fixtures/php-golden/`
- Each fixture set should include a small manifest file with:
  - fixture id,
  - source kind,
  - source refs,
  - source commit,
  - generator command,
  - generator version,
  - input digest,
  - output digest,
  - semantic comparison rules.

## Comparison Rules

- Ignore only documented serialization differences such as object key order when canonicalization is explicitly fixed.
- Do not ignore semantic fields.
- Ignore generated ids or timestamps only when the fixture harness cannot inject fixed values.
- Compare normalized identifiers, digest inputs, approval semantics, snapshot membership, provenance meaning, and audit metadata.

## Generation Order

1. Freeze blueprint contract fixtures for kernel, protocol, workflow, artifacts, provenance, bundles, extensibility, and AI governance.
2. Add a PHP exporter or equivalent generator harness for shared identity, search, deduplication, corpus lock, screening, citation network, full-text, and exports.
3. Build semantic comparators before implementation claims.
4. Only after fixtures exist should implementation gates claim compatibility.

## ADR 0010 Search Fixture Consequences

- Local C# Search fixtures must treat Search output as a raw Search trace, not a deduplicated corpus.
- Duplicate provider sightings and no-id unresolved candidates must remain visible in Search trace fixtures.
- `search-cache-key-include-raw-data-included.json` is required because `ADR 0010` includes `include_raw_data` in local C# cache identity.
- PHP `includeRawData` cache exclusion is an intentional incompatibility until a later ADR reverses it.
- Local Search plan fixtures are schema-closed; PHP-permissive plan behavior belongs only to `search-trace-php-legacy-plan-import.json` and comparator classification.
- CLI/Web display hashes, fallback keys, run files, database rows, job lifecycle rows, audit rows, app snapshots, and app manifests are app projections, not Search fixture authority.
- Future imported-export fixtures are planned by `ADR 0011` and should include `search-import-ris-trace.json`, `search-import-bibtex-trace.json`, `search-import-scopus-csv-trace.json`, `search-import-wos-export-trace.json`, `search-import-zotero-csl-json-trace.json`, `search-import-endnote-export-trace.json`, `search-import-publish-or-perish-csv-trace.json`, `search-import-source-file-digest.json`, `search-import-parser-warning.json`, `search-import-no-id-candidates.json`, `search-import-dedup-not-applied.json`, `search-import-source-specific-id-not-workid.json`, and `search-import-google-scholar-scraping-rejected.json`.
- Imported-export fixtures must bind or preserve raw exported file bytes through `source_file_digest`, keep source-specific ids as source evidence unless a later ADR extends WorkId namespaces, and reject direct Google Scholar scraping.
- Imported-export fixture planning does not authorize Scopus API integration, Web of Science API integration, provider SDKs, credentials, network behavior, import parser implementation, or Deduplication.
- The implemented local fixture set under `fixtures/conformance/search/` is hand-authored and local-only. It does not create PHP compatibility, generated PHP fixture, live-provider, import-parser, Deduplication, Screening, app-authority, or persistence claims.

## Gate 9 Deduplication Contract Consequences

- C# Deduplication implementation is ready to start against `ADR 0012`.
- Dedup fixtures must consume raw Search traces and imported sightings, not live providers and not PHP Search-time deduplicated corpus output.
- Dedup fixtures must preserve raw duplicate sightings before clustering and must not rely on PHP runtime object identity as scientific identity.
- Title fuzzy fixtures must use local default threshold `95` / `0.95` and preserve PHP `92` as drift evidence before any PHP compatibility claim.
- Web membership hashes, persisted dedup runs, representative scoring fallback, and representative-aware snapshots are app projections unless a later ADR admits them as Core contract.
- PHP-generated fixtures are still required before PHP compatibility can be claimed.

- No semantic comparator report format exists yet.
