Status: planning only. No fixtures are generated by this document.

- Repository: `../core`

## Scope

This plan defines the fixture and comparator catalog needed before a C# Search implementation can claim PHP compatibility. It does not generate fixtures and does not implement C# Search.

Search compatibility fixtures must be generated from the pinned PHP source with source commit, generator command, generator version, input digest, output digest, and comparator rules. Existing PHP tests, YAML search-plan fixtures, and VCR cassettes are source evidence, not C# goldens by themselves.

## Negative Fixture Catalog

Required negative cases before implementation claim:

- invalid search term: empty, one character, whitespace-only
- invalid year range: below `1000`, above deterministic upper bound, inverted
- unknown provider alias
- non-positive max results or plan limit
- invalid YAML
- plan root not a mapping
- `searches` or `queries` not a list
- plan item not a mapping
- missing item id
- missing item query/text
- metadata not a mapping
- missing selected plan id
- locked project, if project-lock behavior is in scope
- provider failure captured as partial failure
- all providers failed
- duplicate provider sightings preserved, not deduped
- title-only duplicate rejected as Search identity
- no-id candidate not promoted to canonical identity
- raw-data request satisfied by non-raw cache entry under the local C# cache contract
- PHP raw-data cache ambiguity from excluding `includeRawData` classified as an intentional incompatibility

## Comparator Plan

### General Comparator Rules

- Compare semantic fields, not PHP object identity.
- Ignore generated query ids unless the generator injects fixed ids.
- Ignore runtime duration values except for presence, numeric type, and non-negative shape.
- Ignore exact provider failure exception class names only when the normalized skip reason is explicitly compared.
- Do not ignore provider alias, result count, skip reason, ordering, identifiers, raw-data presence, or cache-key material.
- Do not compare live HTTP response timing.
- Do not make network calls in C# CI.

### Cache Comparator

Compare:

- normalized provider alias set
- provider-order-insensitive hash equality
- provider-different hash inequality
- exact included material fields
- exact excluded material fields
- SHA-256 output string

For field-exclusion fixtures, mutate only one excluded field at a time and require identical cache keys. `include_raw_data` is not an excluded local C# cache field.

### Plan Parser Comparator

Compare:

- item count and order
- root project id
- source name
- item id, label, query, project id, max results, year bounds, providers, raw-data flag, metadata, priority, title/abstract guidance, source index
- selected item ids and priority filters
- override effects
- exact error category and stable message fragment for invalid shapes

### Provider Selection Comparator

Compare:

- active adapter aliases
- validation order: unknown alias before cache lookup and execution
- stats alias order
- success/failure shape
- raw work concatenation order

### Provider Normalization Comparator

Compare:

- normalized identifier namespaces and values after shared identity normalization
- title, year, source provider, abstract, venue, authors, cited count, retraction flag where present
- raw-data presence and digest, not arbitrary object identity
- provider response order

Do not claim provider/network parity until cassette-backed generated fixtures exist.

### Search Trace Comparator

Compare:

- raw provider sightings, including duplicates
- provider alias and provider work id when present
- normalized shared identifiers
- rank/order within provider and overall trace
- raw payload digest when included
- explicit absence of Deduplication output from Search trace

Deduplication output belongs to a later Deduplication comparator.

## Generator Requirements

The PHP fixture generator must:

1. run against commit `b24d0d71ec7b64003465182477e7edb7f49994f4`
2. record exact command and environment
3. force deterministic clocks where PHP behavior depends on current year
4. inject fixed query ids when id equality matters
5. avoid live provider calls except when deliberately regenerating cassette-backed fixtures
6. record input digest and output digest
7. classify every difference as equivalent serialization, intentional incompatibility, PHP defect, C# defect, or unresolved conflict

## Implementation Readiness

Yes, for a local stub-provider Search implementation after `ADR 0010`.

Fixture and comparator design is ready enough for local implementation with explicit non-claims. It is still not ready for PHP compatibility or live provider behavior until:

- generated PHP fixture harness exists
- PHP comparators classify the `includeRawData` cache difference and PHP Search-time deduplication difference
- cassette-backed provider fixture generation is explicitly admitted
- Deduplication and Screening behavior are handled in later gates
- persistence/API/UI/job/cloud behavior is admitted by a later scope
