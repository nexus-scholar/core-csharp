Contributing

Change behavior with evidence

Nexus welcomes contributions that improve the research workflow while preserving reconstructability, authority boundaries, and explicit non-claims.

Good first contribution areas

  • Improve module documentation with source-backed explanations.
  • Add or refine focused tests for accepted contracts.
  • Propose sample block plans for real research workflow situations.
  • Review ADRs for unclear authority or evidence boundaries.
  • Polish renderer surfaces without adding Core dependencies.

Contribution boundaries

  • Do not make AI output a final scientific decision.
  • Do not treat title-only matches, row ids, app hashes, or runtime object identity as scholarly identity.
  • Do not add live providers, credentials, scraping, or production dependencies without an accepted design record.
  • Do not move app projection behavior into Core without an ADR.
  • Do not claim PHP compatibility without fixture or comparator evidence.

Baseline verification

dotnet build NexusScholar.Core.slnx -c Release
dotnet test NexusScholar.Core.slnx -c Release --no-build
dotnet format NexusScholar.Core.slnx --verify-no-changes --no-restore

Public communication standard

  • Say what is implemented and what is only planned.
  • Prefer source links, ADRs, tests, and fixtures over broad claims.
  • Separate Core authority from UI, CLI, Web, and sample behavior.
  • Keep public explanations understandable to researchers, not only library developers.
