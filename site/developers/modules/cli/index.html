Module documentation

CLI

A small local developer and researcher entry point that exercises implemented Core modules and demonstrates the current authority rules without becoming the source of truth for domain behavior.

Scope and ownership

The C# CLI is intentionally small. At the moment it exposes a doctor command for environment and policy reporting and a sample command that walks through a local protocol, workflow, provenance, and bundle-verification path. It is useful as a runnable entry point and smoke surface, not as a replacement for module contracts.

That distinction matters because the broader Nexus ecosystem already has a richer PHP CLI with many user-facing commands. The C# project should not borrow authority from that app-layer surface unless an accepted contract says so.

Status and authority boundary

The current C# CLI is implemented code, but it should be read as integration evidence only. The repo’s own app-recon note for CLI usage says the order of authority remains accepted C# ADRs and fixtures first, pinned PHP Core behavior second, and CLI behavior last.

That is the correct posture here because command surfaces change faster than domain rules. A convenient CLI demo can never be the reason a module contract shifts.

Code surface and dependencies

The entire implementation is visible in Program.cs. That is useful because it makes the command scope obvious. The project currently references Protocol, Workflow, Provenance, Bundles, AI, and Kernel, but the command handlers remain simple and readable.

The doctor path prints platform information and a policy reminder. The sample path uses domain types directly: it creates a protocol draft, approves it, compiles a workflow, appends provenance, creates a bundle manifest, and verifies it.

Related modules and current risks

The CLI touches many modules precisely because it is an orchestration surface. That makes scope discipline important. The main risk is allowing convenience code or app-only output formats to back-propagate into Core module contracts.

For now the module is doing the right thing. It is a thin local harness over implemented domain behavior, not a separate authority layer.
