Extensibility

A narrow contract for plugin identity and capability requests, kept deliberately separate from runtime execution, secrets, and host trust assumptions.

Scope and ownership

Extensibility is intentionally small. It defines what an extension manifest looks like and how requested capabilities are represented, but it does not attempt to be a plugin runtime. That distinction is central to the module’s design quality: the current code is honest about what it knows and equally honest about what it does not yet implement.

The working contract is basically three pieces: an ExtensionManifest with ID, version, entry point, and requested capabilities; an ExtensionCapability enum that names the current capability vocabulary; and a CapabilitySelection record that represents a narrowed grant snapshot. Together they establish a typed capability boundary without pretending to solve sandboxing or process isolation.

Status and evidence

This module is still planning-heavy. There is no dedicated accepted implementation gate for Extensibility yet. The relevant state lives in GATE-00, PORT-MATRIX, and GOLDEN-FIXTURE-PLAN, where the module is still tracked as blocked for a future gate.

The main blocker is not code volume but authority drift. CF-005 and BLUEPRINT-AUDIT both note that the blueprint plugin contract expects richer execution and data-handling semantics than the local PluginContracts sketch currently models. That means the existing C# types are useful local scaffolding, but not yet an accepted conformance surface.

Diagrams

Extensibility is easier to understand when the diagrams make one thing explicit: the current code models request and selection, not plugin execution.

Current contract surface

The existing types define manifest metadata, a capability enum, and a narrowed selection record. No runtime host sits inside this module.

Planned use case boundary

The intended flow is request, narrow, and then hand off to an external host. The module intentionally stops before execution.

Code surface and integration

The implementation is limited to ExtensionManifest.cs, PluginContracts.cs, CapabilitySelection.cs, and the module-local guidance in AGENTS.md. The manifest factory validates non-blank inputs and collapses duplicate capability requests via a set, which is fine for local identity shaping.

The actual integration point today is indirect. Workflow templates can declare capability requirements and optional plugin capability labels, and the workflow compiler validates that capability references are known. But extension manifests are not consulted during compilation, no runtime resolver grants capabilities, and the CLI project reference is only for build alignment rather than actual extension execution.

Current gaps

The largest gap is that the module is contract-only. There is no plugin host, no data-handling policy object, no secrets policy, no process boundary, and no staged output pipeline beyond the caution recorded in the local AGENTS file. The repo is therefore right to avoid making stronger claims than the implementation deserves.

Coverage is also thin compared with the accepted modules. There are architecture tests that include the assembly in dependency checks, but there are no direct unit tests or conformance fixtures for extension manifests, capability grants, or privilege reduction. The planned fixtures plugin-manifest-minimal.json and plugin-permission-grant.json still do not exist, which is a real documentation point, not just a future nicety.

At a glance

  • Status: planning and blocked
  • Primary conflicts: `CF-005`, blueprint drift
  • Role: extension identity and capability requests
  • Next module: AI still needs a real page

Important non-claims

  • No plugin runtime
  • No credential or secret transport
  • No security sandbox claim
  • No conformance fixtures yet
