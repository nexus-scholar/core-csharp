Module documentation

Artifacts

Immutable artifact identity, strict logical path rules, and raw-byte digest verification for portable review bundles and later audit replay.

Scope and ownership

Artifacts is the contract layer that says what a Core artifact is, how it is named, and how its bytes are verified. It is intentionally narrower than storage or export infrastructure. The module does not write to the filesystem, choose transport formats, or manage persistence. Its job is to fix a stable identity envelope around artifact content so other modules can move artifacts around without confusing file paths or containers with scientific identity.

The central object is the artifact descriptor: a bundle of reference name, logical path, kind, media type, size, raw digest, schema identity, and optional protocol, workflow, or provenance binding fields. That mix is what lets Artifacts sit between simple byte hashing and richer bundle reconstruction.

Gate and ADR status

ADR 0009 defines the accepted contract here, and the implemented scope is recorded in GATE-06 with supporting evidence in GATE-06-EVIDENCE. That gate draws an explicit line between local artifact identity and any broader host or ecosystem behavior.

The fixed rules are that bundle manifests use schema nexus.review-bundle.manifest, manifest digests live in the bundle-manifest scope, artifact bytes use the raw-artifact-bytes scope, and logical paths are heavily constrained so a path can serve as a portable reference without turning into an unsafe write target.

Contract map

The descriptor defines artifact identity, while bundle manifest and verifier add transport-safe checking without changing that identity.

Verification use case

The module’s core use case is deterministic bundle checking: inspect manifest, bind known records, verify bytes, then accept or reject the package.

Code surface and behavior

The code is split across the dedicated Artifacts project and the bundle-manifest implementation that consumes it. The main files are src/NexusScholar.Artifacts/Artifacts.cs, ReviewBundleManifest.cs, and BundleVerifier.cs.

The behavior is deliberately strict. Absolute paths, URI-like paths, traversal segments, empty segments, leading or trailing slashes, and backslashes are rejected. Artifact field validation checks blank values, size sanity, and digest shape. Bundle verification then layers manifest identity, required schema checks, protocol and provenance binding checks, artifact byte recomputation, and destructive-overwrite protection on top of the artifact descriptors themselves.

Verification and current limits

The verification story is in good shape for a local gate. BundleTests.cs covers the behavior directly, while BundleFixtureTests.cs replays the conformance fixtures under fixtures/conformance/bundles. Architecture boundary tests also keep the bundle and artifact layers inward-facing rather than letting them depend on Protocol or Provenance assemblies directly.

The remaining limits are explicit and appropriate. Gate 6 does not define replacement policies beyond destructive-overwrite rejection, does not bind archive containers or transport metadata into identity, and does not expose API, UI, or runtime import/export surfaces. The module is therefore strong on local audit-oriented identity, but deliberately incomplete as a full application feature.
