Module documentation

Workflow

Deterministic compilation from approved protocol content into a planned graph with explicit approvals, capability requirements, waivers, and invalidation rules.

Scope and ownership

Workflow translates approved protocol state into a reproducible plan for scientific conduct. Its job is to compile, validate, normalize, and digest the plan. It does not run tasks, host plugins, persist execution state, or call providers. That separation is one of the stronger design decisions in the repo because it stops execution concerns from leaking backward into the planning contract.

The main input is WorkflowCompileInput, which bundles an approved protocol version, a template, compile parameters, schema references, and optional amendment or invalidation material. The output is a digest-bearing WorkflowDefinition with resolved inputs, normalized nodes and edges, approval and capability requirements, declared artifacts, and any invalidation plan entries.

Gate and ADR status

GATE-04 is the accepted local scope for the workflow compiler, with ADR 0005 fixing the template contract and ADR 0006 fixing the compile semantics. The combination matters because Workflow is not just a serializer; it is the point where many earlier contracts are checked together.

The non-negotiable rules are that compilation is explicit-only, approved protocol state is required, template and protocol digests are recomputed and checked, and schema closure is mandatory for the local fixture set. The compiler also normalizes ordering before digesting so that semantically equivalent input order does not produce unstable workflow identity.

The accepted evidence scope is recorded in GATE-04-EVIDENCE. It is important to keep reading this module through that gate boundary because the code intentionally does not claim runtime execution, PHP parity, or blueprint conformance.

Compile use case

Validation pipeline

The strongest part of this module is the validation chain. Before a workflow definition exists, the compiler checks template normalization, amended-protocol witness requirements, protocol digest freshness, schema closure, template digest freshness, graph shape, artifact declarations, approval roles, capability references, waiver mapping, and invalidation notices. This is where the repository turns a broad template concept into a narrow, replayable local contract.

The source is concentrated in Workflow.cs, with the sample entry path in NexusScholar.Cli/Program.cs. That single-file concentration is convenient for inspection, but it also means the module will become harder to navigate as the contract expands. For now, the benefit is that the validation logic is visible in one place and can be read against the gate docs without hunting through multiple assemblies.

Identity, tests, and current limits

Workflow identity is deterministic by design. The workflow ID is derived from protocol and template identity plus compiler identity, and the workflow digest uses a canonical-record envelope over normalized compile output. Template digests and workflow digests are intentionally separated, which helps keep “what was compiled” distinct from “what the resulting plan became.”

Verification is broad. WorkflowCompilerTests.cs covers the behavioral rules directly, and WorkflowFixtureTests.cs replays the fixture contract under fixtures/conformance/workflow/, including positive digested snapshots and negative rejection packs. The remaining limits are explicit: there is still no runtime planner, no AI or plugin execution behavior, and no broader compatibility claim beyond the accepted local gate.
