Module documentation

Search

Raw search-plan, execution-trace, and imported-source contracts that preserve provider evidence and partial-failure details before any deduplication or screening step begins.

Scope and ownership

Search owns executable search inputs and raw execution outputs. That includes SearchPlan, SearchPlanItem, ParsedSearchPlan, SearchTrace, provider attempt and statistics records, raw SearchSighting entries, and imported-export traces parsed from offline files.

The module is deliberately evidence-first. A Search run returns the raw trace of what happened: selected provider aliases, cache identity, per-provider status, and all preserved sightings. It does not collapse that output into a unique corpus, because uniqueness belongs downstream.

Gate and ADR status

The implemented contract is grounded in ADR 0010 and ADR 0011. Those decisions fix two important boundaries: Search emits raw trace evidence, and imported search exports are admitted as source evidence without pretending they are live provider calls.

The most important non-claim here is just as important as the positive contract: Search does not deduplicate, does not convert title-only records into stable identity, and does not treat app projections as authoritative Core behavior.

Code surface and behavior

SearchService.cs validates query shape, result bounds, offsets, and years, normalizes provider aliases, computes cache identity, executes providers, and returns SearchTrace with attempts, stats, sightings, and summary. SearchPlan.cs and SearchPlanParser.cs define the executable-plan side, while SearchImportService and related types preserve imported-export metadata and parser warnings.

One implementation detail worth calling out is that include_raw_data participates in cache identity. That is correct: a trace that strips raw data is not the same trace as one that preserves it.

Related modules and current risks

Search hands directly to Deduplication. Shared identity affects how later modules interpret identifiers, but Search itself should avoid prematurely asserting identity. The main risk is accidental aggregation drift: app consumers often want “unique papers” immediately, and that pressure can erode the raw-trace boundary.

The module should stay conservative. It captures acquisition evidence. It does not silently become corpus construction policy.
