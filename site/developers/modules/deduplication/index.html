Module documentation

Deduplication

Evidence-backed duplicate clustering over raw Search and imported-source sightings, with deterministic representatives and an explicit review boundary for fuzzy or no-id cases.

Scope and ownership

Deduplication owns the move from raw Search evidence to reviewable duplicate structure. Its records model candidates, sighting refs, parser notices, evidence edges, clusters, representative projections, unresolved candidates, and review-required pairs. The center of the implemented surface is DeduplicationService.Execute(...).

The module does not invent identity from titles. It consumes Search traces and import traces, preserves those bindings, forms automatic clusters only through accepted exact identifier evidence, and records fuzzy-title and source-specific overlap as review-needed evidence.

Gate and ADR status

The local contract is fixed by ADR 0012 and the Gate 9 Dedup evidence pack. This is implemented local C# behavior, but it deliberately stops short of broader PHP compatibility claims. The accepted contract resolves important questions such as input shape, transitive cluster formation, representative election, and the default fuzzy title threshold.

One concrete decision matters for anyone reading older PHP material: the local default fuzzy title threshold is 0.95. The older 92 drift is treated as drift, not adopted as the C# default.

Class map

Use case flow

The pipeline is raw sightings in, exact-id clusters when justified, review candidates when not, and a deterministic result that retains the evidence graph.

Code surface and implemented behavior

DeduplicationModels.cs defines the result vocabulary. DeduplicationService.cs builds candidates from SearchTrace and SearchImportTrace, records source evidence, unions exact identifier overlaps, computes title similarity for review candidates, elects a representative deterministically, and emits a nexus.deduplication.result record with policy id, threshold, clusters, unresolved candidates, warnings, and non-claims.

The current provider-priority table and representative scoring are explicit in code. That is a strength, because representative election is exactly the kind of behavior that should be stable and auditable rather than implicit in collection order.

Related modules and current risks

Deduplication sits between Search and Screening, and it leans on Shared for normalized stable identifiers. The main risk is silent identity inflation: once a cluster exists, downstream code may be tempted to forget the raw members or act as if fuzzy similarity created scientific identity. The current contract resists that by preserving all direct evidence and keeping review-required edges visible.

That is the right boundary. Deduplication can form reviewable structure; it cannot convert ambiguous evidence into final scientific truth.
