Avalonia Blocks

A renderer-only prototype that consumes UiContracts workspace plans and turns them into Avalonia views without pulling Core domain projects into the desktop layer.

Scope and ownership

Avalonia Blocks is not the desktop product. It is a renderer prototype for WorkspacePlan JSON. Its job is to prove that the contract can be consumed by a concrete UI without contaminating Core. That is why the current surface is view models and user controls, not app services, persistence, or domain commands.

The module exposes controls such as WorkspacePlanView, ResearchBlockView, EvidenceRefListView, ValidationRefListView, BlockActionListView, and PayloadJsonView. Actions are callbacks only. They are display plumbing, not Core mutation.

Boundary status

The active design reference is AVALONIA-RENDERER-PROTOTYPE-v0. The document is explicit about what this module is not: no shell, no Core state loading, no app-state persistence, no AI execution, and no command bridge into the domain stack.

That boundary is healthy. Desktop renderer work is valuable here only if it remains a consumer of contracts rather than a backdoor into domain behavior.

Diagrams

Class map

The renderer projects a contract into view models and controls. The domain stack stays outside the module.

Use case flow

The supported path is one-way: deserialize contract, project view models, render, optionally invoke caller-supplied callbacks.

Code surface and dependencies

The implementation is split between view-model projection in WorkspacePlanViewModels.cs and a small set of controls such as WorkspacePlanView and ResearchBlockView. The project should reference only NexusScholar.UiContracts and Avalonia packages.

That dependency line is not cosmetic. If this renderer starts referencing Kernel, Protocol, Search, or Screening directly, it will collapse the separation that the UI prep work was trying to establish.

Related modules and current risks

Avalonia Blocks depends on UiContracts and on sample plan data. It should not depend on Core modules. The main risk is accidental app-shell growth inside the renderer project: once callbacks exist, it is easy to start wiring direct command execution and domain reads here.

For now the module stays disciplined. It proves renderer feasibility and nothing more.

At a glance

  • Status: renderer prototype
  • Authority: non-authoritative sample consumer
  • Depends on: `NexusScholar.UiContracts`
  • Must not depend on Core domain projects
