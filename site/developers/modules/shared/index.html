Module documentation

Shared

Stable scholarly identity primitives and corpus membership rules that let later modules talk about the same work without relying on titles, object identity, or app-specific fallback hashes.

Scope and ownership

Shared owns the local scientific identity surface: approved namespaces, strict <namespace>:<value> parsing, namespace-specific normalization, stable ordering, overlap checks, and the distinction between identified works and unresolved candidates. The main types are WorkIdNamespace, WorkId, WorkIdSet, ScholarlyWork, and CorpusSlice.

This module matters because many later modules need to speak about “the same work” without inventing their own heuristics. Shared is where the repo draws the hard line that title-only equality is not scientific identity.

Gate and ADR status

The accepted boundary comes from ADR 0007 and the local Gate 9 shared-identity work. The current implementation is intentionally local in scope. It resolves how C# Core handles stable scholarly identifiers without claiming that surrounding applications or PHP projections are canonical.

The approved namespaces are explicit in code: doi, arxiv, openalex, s2, pubmed, pmcid, ieee, doaj, and internal. Namespace text outside that list is rejected, not silently stored.

Code surface and dependencies

The main implementation lives in SharedIdentity.cs. The dependency surface is intentionally tiny: the project depends on Kernel only. That is exactly right for a foundational identity module.

A few implementation details are especially important. DOI and arXiv values are normalized by namespace-specific prefix stripping, work ids are deduplicated and sorted by precedence, identified works require at least one stable identifier, and unresolved candidates can exist but cannot silently become immutable corpus members without accepted evidence.

Related modules and current risks

Shared is upstream of Search, Deduplication, Bundles, and Screening. The main risk is pressure to admit app-local fallback keys, titles, or runtime object identity into this module. That would immediately weaken every downstream claim about identity.

The current implementation holds the line correctly. If a record lacks a stable identifier, it stays unresolved.
