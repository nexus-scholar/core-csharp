Bundles

Portable review-bundle manifests and verification rules that let the system move scientific state as a package without turning local files or app projections into authority.

Scope and ownership

Bundles owns the manifest contract for review packages. The implementation is centered on ReviewBundleManifest, its binding records for protocol, workflow, provenance, artifact membership, shared-identity membership, and unresolved candidates, plus the verifier that checks whether a manifest is structurally safe to import or compare.

The important point is that Bundles does not transport arbitrary application state. It packages references to already-governed records. Artifact paths remain paths, digests remain the scientific identity, and the manifest records what a bundle claims to contain so a verifier can reject malformed, stale, or destructive inputs.

Gate and ADR status

The local C# behavior is the Gate 6 bundle surface, bounded by ADR 0009 and the Gate 6 evidence pack. That accepted scope is narrower than a full import/export product. It defines a review-bundle manifest, required sections, bundle-safe overwrite rules, digest expectations, and verification findings.

This module is implemented. The current verifier emits findings such as invalid-manifest, missing-required-section, checksum-mismatch, stale-manifest-digest, and destructive-overwrite. Those categories matter because the bundle contract is mostly about refusal rules.

Code surface and dependencies

The implemented surface lives in ReviewBundleManifest.cs and BundleVerifier.cs. The project depends only on NexusScholar.Kernel and NexusScholar.Shared. That is the correct shape: the module references stable identity and digest machinery, but it does not absorb the full higher-level domain graph.

The current class surface also makes the module’s intent obvious. Manifest records describe sections; verifier options tune checks; verification results return structured findings. There is no hidden import side effect. The module tells the caller what is wrong and leaves import policy to a later layer.

Current limits and related modules

Bundles is downstream from Protocol, Workflow, Provenance, Artifacts, and Shared. It assumes those modules already own the records being referenced. It does not redefine them. The obvious risk is scope creep: app-specific full-text manifests, zip transport details, and persistence concerns are easy to drag into this module, but that would weaken the contract.

The local implementation is intentionally conservative. It verifies a review-bundle manifest. It does not claim hosted transport, remote trust negotiation, or CLI and Web manifest compatibility beyond the accepted contract.
