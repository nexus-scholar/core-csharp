Module documentation

AI

Governed AI task and proposal primitives that preserve the repository rule that model output is suggestion or bounded transformation until an authorized human accepts it.

Scope and ownership

The AI module does not implement a complete AI execution subsystem. What it owns today is the policy boundary and proposal shape. AiTaskPolicy classifies authority, records whether human approval is required, and rejects unsafe policy combinations. AiProposal<T> and AcceptedAiProposal<T> separate raw model output from accepted output.

That split matters because this repository treats AI as one input to scientific work, not the authority over it. The local code surface is therefore small on purpose. It encodes guardrails before richer provider, logging, or workflow integration exists.

Status and contract boundary

This module is partially implemented. The contract surface exists in code, but the larger AI governance gate remains unresolved for later work. In particular, conflict notes around AI governance were deferred rather than silently patched over. So this page documents the implemented policy surface, not a completed accepted gate.

The most important current rule is visible in the authority vocabulary itself: ReadOnlySuggestion, BoundedTransformation, ScientificDecisionProposal, and ExternalActionProposal. Higher-authority categories must require human approval. If a caller tries to create a high-authority policy without that requirement, AiTaskPolicy.Create(...) refuses it.

Diagrams

Use case flow

The expected path is simple: define authority, capture model output as a proposal, and require acceptance before any downstream scientific use.

Code surface and dependencies

The current code lives in AiTaskPolicy.cs and AiProposal.cs. The project depends on NexusScholar.Kernel and NexusScholar.Provenance. That dependency shape is appropriate for recording bounded proposals and future acceptance evidence without letting AI reach outward into app frameworks or provider SDKs.

The module-local AGENTS.md sharpens the same rule in prose: model output is never canonical scientific truth. That is consistent with the repository product laws and means later integration work should preserve, not bypass, this contract.

Current gaps and related modules

The obvious missing pieces are intentional: no provider abstraction, no prompt/result persistence contract, no full provenance acceptance path, no workflow binding, and no completed gate for broader AI governance. The implemented surface is useful because it stops callers from pretending those missing pieces are solved.

AI is closely related to Workflow, Provenance, Screening, and UiContracts. It will eventually show up inside those modules as proposal evidence, not as silent authority.
